## はじめに

**この記事の対象読者**
- Claude Code、Claude.ai、Claude APIを業務で活用している開発者
- AIに同じ指示やガイドラインを何度も繰り返し説明している人
- 組織固有のワークフローやブランドガイドラインをAIに適用したい人

**この記事でわかること**
- Claudeのカスタムスキル（Custom Skills）の基本概念と利点
- カスタムスキルの作成方法（ファイル構成、メタデータ、内容の書き方）
- 効果的なスキル設計のベストプラクティス
- 実際の活用例とユースケース

**この記事で扱わないこと**
- Claudeの基本的な使い方
- プロンプトエンジニアリングの一般論
- API経由での高度なスキル管理

## 「毎回同じことを説明する」ストレス

Claudeを業務で活用していると、こんな経験はありませんか？

- 週次レポートの形式を毎回説明している
- ブランドガイドライン（ロゴ、カラー、フォント）を毎回貼り付けている
- コードレビューの観点を毎回列挙している
- 顧客フィードバックの分析方法を毎回指示している

これらは一度確立したワークフローなのに、新しい会話のたびに同じ説明を繰り返す必要があります。カスタムインストラクション（Custom Instructions）では文字数制限があり、プロジェクト機能は特定のプロジェクトにしか適用されません。

**Claudeのカスタムスキル（Custom Skills）は、この問題を解決します。**

カスタムスキルを使えば、あなたの確立したワークフローや組織のガイドラインを一度パッケージ化するだけで、Claudeが自動的に適切な場面で適用してくれます。

## カスタムスキルとは

カスタムスキルは、Claudeにあなた独自の作業方法や基準を教えるための再利用可能なパッケージです。

### 他の機能との違い

| 機能 | 用途 | スキルとの違い |
|------|------|---------------|
| **プロジェクト** | 特定プロジェクトの長期的なコンテキスト蓄積 | スキルはすべての会話で自動適用 |
| **カスタムインストラクション** | 全体的な好みや方針の設定 | スキルは特定タスクの詳細な手順を含む |
| **通常のプロンプト** | 一回限りのタスク | スキルは繰り返し使う作業を自動化 |

### カスタムスキルが有効なシーン

カスタムスキルは次の3つのシーンで特に効果を発揮します：

**1. 洗練されたアプローチがある場合**
- 経験から確立したプロセス（レポート構成、分析手法など）
- 毎回説明し直すのが面倒な手順

**2. 特定の資料が必要な作業**
- ブランドガイドライン、テンプレート、ドメイン知識
- 一貫して適用すべき基準

**3. 複数の要素を組み合わせるセットアップ**
- 複数ファイルの参照が必要
- 異なる要件の調整が必要

## カスタムスキルの作成方法

### 基本構成

カスタムスキルは次の構造を持つZIPファイルとして作成します：

```
my-skill.zip
└── my-skill/
    ├── Skill.md         # 必須: スキルの本体
    ├── REFERENCE.md     # 任意: 参考資料
    └── resources/       # 任意: その他のファイル
        ├── script.py
        └── template.json
```

**重要**: ZIPファイルの直下にスキルフォルダを配置してください。余分な親フォルダを作らないように注意が必要です。

### Skill.mdの構造

すべてのスキルは`Skill.md`ファイルから始まります。このファイルはYAMLフロントマターで始まる必要があります。

```yaml
---
name: "Brand Guidelines"
description: "Apply Acme Corp brand guidelines to presentations and documents, including official colors, fonts, and logo usage."
version: "1.0.0"
dependencies: "python>=3.8, pandas>=1.5.0"
---

# スキルの内容（Markdown形式）

## 概要
このスキルは...

## 使用方法
1. ...
2. ...

## 詳細な手順
...
```

#### 必須メタデータ

**name（名前）**
- 人間が読みやすい識別子
- 最大64文字
- 動名詞形（gerund form）を使用: `processing-pdfs`、`analyzing-spreadsheets`
- 曖昧な名前（`helper`、`utils`）は避ける

**description（説明）**
- スキルの目的と使用場面を明確化
- 最大200文字
- 三人称で記述
- 具体的に何をするか、いつ使うかを明記

説明文の良い例：
```yaml
description: "週次チームレポートを作成します。成果、ブロッカー、優先事項を構造化して整理します。"
```

悪い例：
```yaml
description: "レポートを作る" # 曖昧すぎる
```

#### 任意メタデータ

**version**: バージョン管理（例: `1.0.0`）
**dependencies**: 必要なソフトウェアパッケージ（例: `python>=3.8, pandas>=1.5.0`）

### スキルの内容を書く

Skill.mdのMarkdown本体には、指示と詳細を記述します。

#### 簡潔さを保つ

**重要**: Skill.mdの本体は500行以下に抑えてください。

理由：
- Claudeはコンテキストウィンドウを会話履歴や他のスキルと共有している
- すべてのトークンがコストに影響する
- Claudeは既に十分賢いので、本当に必要な情報だけを追加する

長くなる場合は、追加ファイル（`REFERENCE.md`など）に分割し、必要に応じて参照する「Progressive Disclosure（段階的開示）」パターンを使います。

#### 自由度のレベルを調整する

タスクの性質に応じて、指示の具体性を調整します：

| 自由度 | 適用場面 | 記述方法 |
|--------|----------|----------|
| **高** | 柔軟な判断が必要 | テキスト指示 |
| **中** | 推奨パターンがある | 疑似コード、パラメータ |
| **低** | エラーが起きやすい操作 | 具体的なスクリプト |

#### ワークフローはチェックリスト形式で

複雑な操作は明確なステップに分解し、コピー可能なチェックリストにします。これによりClaudeが進捗を追跡できます。

```markdown
## レポート作成の手順

- [ ] 先週の成果を箇条書きで抽出
- [ ] ブロッカーを重要度順に整理
- [ ] 今週の優先事項トップ3を特定
- [ ] テンプレート形式でフォーマット
- [ ] 最終チェック（誤字脱字、リンク確認）
```

#### フィードバックループを組み込む

検証パターンを実装すると、出力品質が大幅に向上します：

```markdown
## 検証プロセス

1. バリデータを実行
2. エラーを修正
3. ステップ1に戻る（エラーがなくなるまで）
```

### コードとスクリプトの扱い

#### ユーティリティスクリプトを提供する

信頼性とトークン効率のため、事前に作成したスクリプトを含めることができます。

```markdown
## データ変換スクリプト

`resources/transform.py`スクリプトを実行してデータを変換してください：

\`\`\`bash
python resources/transform.py input.csv output.json
\`\`\`

このスクリプトは以下の処理を行います：
- CSVデータの読み込み
- 必須フィールドの検証
- JSON形式への変換
```

Claudeがスクリプトを実行すべきか、参考として読むべきかを明確にしてください。

#### エラー処理を明示的に

問題をClaudeに丸投げせず、明示的に処理します：

```python
# Good: エラーハンドリングが明確
try:
    result = process_data(input_file)
except FileNotFoundError:
    print(f"Error: {input_file} not found. Please check the file path.")
    sys.exit(1)
except ValueError as e:
    print(f"Error: Invalid data format - {e}")
    sys.exit(1)
```

#### 検証可能な出力

バッチ操作では、plan-validate-executeパターンを使用します：

1. 変更計画を中間JSONファイルに出力
2. スクリプトで計画を検証
3. 問題なければ変更を適用

```markdown
## バッチリネームの手順

1. リネーム計画を作成: `rename_plan.json`
2. 検証スクリプトを実行: `python validate_rename.py rename_plan.json`
3. 検証が通ったら実行: `python execute_rename.py rename_plan.json`
```

### アンチパターン（避けるべきこと）

以下は避けるべき設計パターンです：

- ❌ Windowsスタイルのバックスラッシュパス（常にスラッシュを使用）
- ❌ 明確なデフォルトなしに過剰な選択肢を提供
- ❌ 深くネストされたファイル参照（完全に読み込まれない原因）
- ❌ ツールやパッケージが事前インストールされていると仮定
- ❌ 時間依存の情報（「2024年現在では...」は避け、「旧パターン」セクションで非推奨機能を扱う）
- ❌ 用語の混在（「APIエンドポイント」「URL」「ルート」を混ぜて使わない）

### テストとイテレーション

#### アップロード前のチェック

- 明確さを確認
- ファイル参照を検証
- サンプルプロンプトでテスト

#### アップロード後のテスト

1. `Settings > Capabilities`でスキルを有効化
2. さまざまなプロンプトを試す
3. Claudeの推論を確認し、適切に呼び出されているか確認

#### 複数モデルでテスト

スキルをデプロイする前に、Claude Haiku、Sonnet、Opusで動作確認してください。モデルの能力によって効果が異なります。

#### 評価を先に構築

広範なドキュメントを書く前に、実際のギャップを特定するテストシナリオを作成します。スキルありとなしでパフォーマンスを測定します。

#### Claudeと反復的に開発

あるClaudeインスタンスにスキル設計を手伝ってもらい、別のインスタンスで実際のタスクを実行してテストします。これにより仮定と実際のギャップが明らかになります。

## 実際の活用例

カスタムスキルは様々な場面で活用できます：

### 1. 週次チームレポート

```yaml
---
name: "Weekly Team Update"
description: "Create structured weekly team updates with wins, blockers, and priorities in company format."
---

## フォーマット

週次レポートは以下の構成で作成してください：

### 📊 今週の成果（Wins）
- 完了した主要タスク
- 達成したマイルストーン
- チームの成果

### 🚧 ブロッカー（Blockers）
- 進行を妨げている課題
- 必要なサポートや決定事項

### 🎯 来週の優先事項（Priorities）
1. 最優先タスク
2. 重要度2位
3. 重要度3位
```

### 2. 顧客フィードバック分析

```yaml
---
name: "Customer Feedback Analysis"
description: "Analyze customer feedback, categorize issues, and identify patterns with actionable insights."
---

## 分析プロセス

1. フィードバックを以下のカテゴリに分類：
   - 機能リクエスト
   - バグ報告
   - UX/UI改善提案
   - パフォーマンス問題
   - ドキュメント関連

2. 各カテゴリ内でパターンを特定：
   - 頻度の高い要望
   - 重要度の高い問題
   - 影響範囲の大きさ

3. アクションアイテムを優先度順に整理
```

### 3. ブランドコンプライアンス

```yaml
---
name: "Brand Guidelines Checker"
description: "Ensure documents comply with company brand guidelines for logos, colors, fonts, and templates."
---

## ブランド仕様

### カラーパレット
- プライマリ: #0066CC
- セカンダリ: #FF6B35
- アクセント: #4ECDC4

### フォント
- 見出し: Montserrat Bold
- 本文: Open Sans Regular

### ロゴ使用規則
- 最小サイズ: 40px高さ
- クリアスペース: ロゴ高さの25%
- 背景: 白または#F5F5F5のみ

## チェック項目
- [ ] カラーパレットが仕様に準拠
- [ ] フォントが正しく使用されている
- [ ] ロゴのクリアスペースが確保されている
```

### 4. 技術ドキュメント

```yaml
---
name: "Technical Documentation Standards"
description: "Apply company technical writing standards with proper terminology and notation."
---

## 用語統一

| 使用する用語 | 避ける用語 |
|-------------|-----------|
| API | api, Api |
| JavaScript | JS, javascript |
| データベース | DB, database |

## ドキュメント構成

1. 概要
2. 前提条件
3. セットアップ手順
4. 使用例
5. トラブルシューティング
6. 参考リンク
```

## 品質チェックリスト

スキルを完成させる前に確認すべき項目：

- [ ] 具体的な説明とキーワード、使用タイミングを記載
- [ ] Skill.mdは500行以下で、必要に応じて別ファイルに分割
- [ ] 時間依存の内容を適切に処理
- [ ] 全体で用語を統一
- [ ] 抽象的でなく具体的な例を使用
- [ ] 検証を含む明確なワークフローステップ
- [ ] 明示的なエラーハンドリングを含むスクリプト
- [ ] 複数のモデルと実際のシナリオでテスト済み

## スキルのアップロードと管理

### アップロード方法

1. スキルフォルダをZIP形式で圧縮
2. Claude.aiの`Settings > Capabilities > Skills`にアクセス
3. カスタムスキルをアップロード
4. スキルを有効化

### 非技術者向け

技術的な知識がなくても、Claudeに手伝ってもらってスキルファイルを作成できます：

```
「週次レポートを作成するスキルを作りたいです。
以下の形式で作成したいです：
- 今週の成果
- ブロッカー
- 来週の優先事項

これをSkill.mdファイルにしてください」
```

Claudeが適切なフォーマットで作成してくれます。

## おわりに

Claudeのカスタムスキルは、**「毎回同じ説明をする問題」を解決する**強力な機能です。

一度ワークフローをパッケージ化すれば、すべての会話でClaudeが自動的に適用してくれます。これにより：

- 説明の繰り返しから解放される
- 組織のガイドラインを一貫して適用できる
- 確立したベストプラクティスを再利用できる
- チーム全体で同じ基準を共有できる

### 今すぐ始めるには

1. 毎回繰り返している説明や手順を特定する
2. それをSkill.mdファイルにまとめる
3. ZIP形式で圧縮してClaudeにアップロード
4. 実際のタスクで試して改善を繰り返す
